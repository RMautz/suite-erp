# Diseño: Plan 11 — Vertical Transporte (flota, tarifario, órdenes de entrega y proformas)

**Fecha:** 2026-07-15
**Estado:** Aprobado por el usuario; re-modelado sobre su proforma REAL (PF-000004); verificación adversarial rondas 1-3 aplicadas (21 + 12 + 16 hallazgos)
**Fase:** 2 del roadmap (verticalización). El usuario opera carga consolidada: su empresa será el primer cliente real Y la vertical queda como oferta del SaaS.

## 1. Propósito

Replicar el flujo comercial REAL del transportista de carga: registrar **órdenes de entrega (ODE)** — carga que un cliente encarga llevar a un destino, con bultos/kilos/M3 y neto tarifado por kilo afecto —, agruparlas por cliente en una **PROFORMA numerada** (PF-000004) que se envía para visto bueno, y al aprobarse convertirla en factura SII por el flujo existente. Módulo activable por empresa; comercio y transporte conviven.

## 2. Decisiones fundacionales (aprobadas por el usuario; calibradas contra su proforma real)

| Decisión | Elección |
|---|---|
| Unidad comercial | **ODE (orden de entrega)**: cliente, fecha ingreso, destino, guías (docum), OC del cliente, bultos, kilos, M3, **kilo afecto** derivado, neto. El reparto/viaje como agrupador logístico queda fuera de v1 |
| Kilo afecto | `kilo_afecto = round(max(kilos, m3 × factor_volumetrico))` calculado SIEMPRE server-side — verificado contra su proforma: max(175, 1.26×250)=315; max(787, 2.88×250)=787. **Factor configurable por empresa** (`empresas.factor_volumetrico`, default 250, `check between 1 and 10000`). Si el cálculo da < 1 kg → error claro (`'La carga es demasiado pequeña para tarifar'`) |
| Neto de la ODE | **Tarifario por destino + editable**: maestro `destinos` con $/kg afecto; la UI pre-calcula `kilo_afecto × tarifa_kg` y el usuario puede ajustar. El neto que persiste lo manda el llamador (patrón precios de cotizaciones); el tarifario es sugerencia con control por VISIBILIDAD (la nómina muestra kilo afecto y neto lado a lado) |
| Prospectividad | Cambios de `tarifa_kg` y `factor_volumetrico` son PROSPECTIVOS: `kilo_afecto` y `neto` ya persistidos jamás se recalculan (pgTAP lo fija) |
| IVA | Por ODE solo display; el IVA REAL de proforma/factura = `round(neto_total × 0.19)::integer` — verificado contra su sistema (43.289 = round(227.836×0.19) ≠ Σ IVAs por línea). Flete de carga AFECTO; `exenta=false` |
| Proforma | **Documento propio con ciclo completo**: correlativo (display `PF-000004`), estados, vista imprimible NO tributaria con NÓMINA y TOTALES (bultos/M3/kilo afecto derivados de las ODEs; neto/IVA/total almacenados), conversión a factura al aprobarse. Nro/Fecha FAV de su pantalla = nuestra factura enlazada |
| Ciclo | ODEs `registrada` de UN cliente → `crear_proforma` (reserva) → `enviada` → `aprobada` → `facturar_proforma` → nota de venta borrador → emitir. **`rechazada` desde borrador, enviada Y aprobada** (motivo; LIBERA las ODEs — el detalle de una rechazada muestra solo encabezado+motivo+totales almacenados, SIN nómina ni Imprimir: las ODEs ya son de otros). Sin edición: se rechaza y se re-crea |
| Reversa | **`anular_estado_pago`**: nota de venta `borrador` sin folio O **DTE `rechazado` por el SII** (folio consumido = gap justificable). Documento y líneas se eliminan, ODEs vuelven a `registrada` (conservando proforma), proforma vuelve a `aprobada`. Con factura ACEPTADA: NC, nada se libera (v1) |
| Flota / despacho | Maestros `vehiculos`/`conductores`; la ODE lleva vehículo/conductor OPCIONALES — al crear O después vía **`asignar_despacho`** (RPC mínima: la asignación real ocurre al despachar; alimenta la rentabilidad del Plan 12). No salen en la proforma |
| Activación | `empresas.modulo_transporte` (toggle en Configuración, dueno/admin); NAV muestra Flota/Tarifario/Entregas/Proformas solo si activo. `crear_orden_entrega` y `crear_proforma` EXIGEN módulo activo; el resto funciona con módulo off. Apagar solo oculta UI |

## 3. Arquitectura y modelo de datos (migración 0016)

Convenciones vigentes: `empresa_id` + RLS + grants explícitos + FK compuestas + `unique (empresa_id, id)` en tablas referenciadas por FK compuesta; CLP entero; kilos/M3 `numeric(9,2)`; español.

- **`empresas`** — `modulo_transporte boolean not null default false` y `factor_volumetrico integer not null default 250 check (factor_volumetrico between 1 and 10000)` (ALTER). **Endurecimiento pre-existente**: `revoke update on public.empresas from authenticated; grant update (razon_social, giro, direccion, comuna, giro_emisor, direccion_emisor, comuna_emisor, resolucion_sii_numero, resolucion_sii_fecha, certificado_cifrado, certificado_password_cifrada, dte_api_key_cifrada, modulo_transporte, factor_volumetrico) to authenticated` — `id`/`rut`/`organizacion_id` FUERA. pgTAP: update de `organizacion_id` por dueno → 42501.
- **`vehiculos`** — `patente text not null check (patente ~ '^[A-Z]{4}[0-9]{2}$' or patente ~ '^[A-Z]{2}[0-9]{4}$')`; `unique (empresa_id, patente)`; `descripcion` nullable, `capacidad_kg integer` nullable `check (> 0)`, `activo not null default true`, `unique (empresa_id, id)`. Select miembros; insert/update dueno/admin; `grant update (descripcion, capacidad_kg, activo)`. 23505 → `'Ya existe un vehículo con esa patente'`.
- **`conductores`** — `rut text not null check (rut = app.normalizar_rut(rut) and app.validar_rut(rut))`; `unique (empresa_id, rut)`; `nombre not null`, `telefono` nullable, `activo not null default true`, `unique (empresa_id, id)`. Como vehiculos; `grant update (nombre, telefono, activo)`. 23505 → `'Ya existe un conductor con ese RUT'`.
- **`destinos`** (tarifario) — `nombre text not null`, **SOLO índice único CI** `(empresa_id, lower(nombre))` (sin unique plano: el CI lo implica — lección 9b sin la redundancia), `tarifa_kg integer not null check (tarifa_kg > 0)`, `activo not null default true`, `unique (empresa_id, id)`. Insert/update directo dueno/admin (tarifas = dinero; vendedor NO edita); `grant update (tarifa_kg, activo)` — nombre/empresa_id inmutables (la colisión CI solo puede ocurrir en insert: un solo punto de mapeo). 23505 → `'Ese destino ya existe'`.
- **`proformas`** (antes que ordenes_entrega) — `numero integer not null` (correlativo; display `'PF-' || lpad(numero, 6, '0')`), `unique (empresa_id, numero)`, `cliente_id` NOT NULL (FK compuesta), `fecha date not null default current_date`, `neto`/`iva`/`total` integers `>= 0` `check (total = neto + iva)` (ALMACENADOS), `estado` (`check in ('borrador','enviada','aprobada','rechazada','facturada')` default `'borrador'`), `motivo_rechazo`, `notas`, `documento_venta_id` (FK compuesta NULLABLE), `unique (empresa_id, id)`. Índice `(empresa_id, estado, creado_en desc)`. CERO escritura directa. **Máquina**: `borrador → enviada | rechazada`; `enviada → aprobada | rechazada`; `aprobada → rechazada | facturada` (facturada SOLO vía `facturar_proforma`); `facturada → aprobada` (SOLO vía `anular_estado_pago`). Todo rechazo libera ODEs.
- **`ordenes_entrega`** — `numero integer not null` (correlativo, `unique (empresa_id, numero)`), `cliente_id` NOT NULL (FK compuesta), `fecha_ingreso date not null`, `destino_id` NOT NULL (FK compuesta), `docum text` nullable (guías, texto libre), `oc_cliente text` nullable, `bultos integer not null check (bultos > 0)`, `kilos numeric(9,2) not null check (kilos > 0)`, `m3 numeric(9,2)` nullable `check (m3 >= 0)`, `kilo_afecto integer not null check (kilo_afecto > 0)` (CALCULADO server-side), `neto integer not null check (neto > 0)` (del llamador), `vehiculo_id`/`conductor_id` NULLABLES (FK compuestas), `notas` nullable, `estado` (`check in ('registrada','facturada','anulada')` default `'registrada'`), `motivo_anulacion` nullable, `proforma_id` (FK compuesta NULLABLE), `documento_venta_id` (FK compuesta NULLABLE), `creado_en`, `unique (empresa_id, id)`. Índices: `(empresa_id, estado, fecha_ingreso desc)`, `(empresa_id, cliente_id, estado)`, parciales de `proforma_id` y `documento_venta_id`. CERO escritura directa.
- **Vista `ventas_por_producto`** (0012): recreada con `where l.producto_id is not null`.

## 4. RPCs (security definer, `set search_path = public`, revoke anon/public + grant authenticated, validan rol via `app.tiene_rol_en_empresa`)

Roles: todo el ciclo = `dueno`/`admin`/`vendedor`; `contador`/`bodeguero` solo leen. Normalización: `p_notas`/`p_motivo`/`p_docum`/`p_oc` con `nullif(trim(coalesce(..., '')), '')`. **Regla anti-TOCTOU (contrato)**: TODA validación de estado/pertenencia corre DESPUÉS del `for update`; TODA operación multi-ODE toma locks `order by id for update` antes de cualquier UPDATE masivo. Garantizadas por diseño: crear_proforma vs crear_proforma sobre la misma ODE, anular vs reservar, facturar vs rechazar.

- **`crear_orden_entrega(p_empresa, p_cliente, p_fecha date, p_destino uuid, p_docum text, p_oc text, p_bultos integer, p_kilos numeric, p_m3 numeric, p_neto integer, p_vehiculo uuid, p_conductor uuid, p_notas text) returns uuid`** — rol → `'Tu rol no permite registrar órdenes de entrega'`; exige módulo activo → `'El módulo de transporte no está activo'`. Valida: cliente activo → `'Cliente no válido'`; destino activo → `'Destino no válido'`; `p_fecha is null` → `'Ingresa la fecha de ingreso'`; `p_bultos is null or < 1` → `'Los bultos deben ser un entero mayor a 0'`; **`v_kilos := round(p_kilos, 2)`** y `v_kilos is null or <= 0` → `'Los kilos deben ser mayores a 0'`; `v_kilos > 1000000` → `'Los kilos superan el máximo admitido'`; **`v_m3 := round(p_m3, 2)`**, `v_m3 < 0` → `'Los M3 no pueden ser negativos'`, `v_m3 > 100000` → `'Los M3 superan el máximo admitido'` (con factor ≤ 10000 el producto cabe siempre en int4); `p_neto is null or < 1` → `'El neto debe ser un entero mayor a 0'`; vehículo/conductor SI vienen → activos de la empresa → `'Vehículo no válido'`/`'Conductor no válido'`. **Server-side**: `v_kilo_afecto := round(greatest(v_kilos, coalesce(v_m3, 0) * v_factor))::integer` (la RPC ni recibe kilo_afecto); si `v_kilo_afecto < 1` → `'La carga es demasiado pequeña para tarifar'`. Correlativo DENTRO de `pg_advisory_xact_lock(hashtextextended('ordenes_entrega:' || p_empresa::text, 42))`.
- **`asignar_despacho(p_empresa, p_orden, p_vehiculo uuid, p_conductor uuid)`** — rol → `'Tu rol no permite asignar despachos'`. Lookup for update; 0 filas → `'La orden de entrega no existe'`; exige no-anulada → `'No se puede asignar despacho a una orden anulada'` (registrada O facturada valen: el despacho físico es independiente de la facturación). Vehículo/conductor NULLABLES (null = desasignar); si vienen → activos de la empresa → `'Vehículo no válido'`/`'Conductor no válido'`.
- **`anular_orden_entrega(p_empresa, p_orden, p_motivo)`** — rol → `'Tu rol no permite anular órdenes de entrega'`. Lookup for update; 0 filas → `'La orden de entrega no existe'`; motivo no-whitespace → `'La anulación requiere un motivo'`; orden de checks FIJADO: primero estado (`'Solo se puede anular una orden registrada'`), después reserva (`'La orden está en una proforma; recházala primero'`).
- **`crear_proforma(p_empresa, p_cliente, p_ordenes uuid[], p_notas) returns uuid`** — rol → `'Tu rol no permite gestionar proformas'`; exige módulo activo → `'El módulo de transporte no está activo'`. Vacío (`coalesce(array_length,0) < 1`) → `'La proforma necesita al menos una orden de entrega'`. Dedup `array_agg(distinct)`. Cliente activo → `'Cliente no válido'`. Bloqueo `order by id for update`; count ≠ cardinalidad dedup, o no-`registrada`, o cliente distinto, o reservada → `'Hay una orden no válida, de otro cliente, ya facturada o en otra proforma'`. Dinero: `v_neto bigint := Σ netos`; **umbral EXACTO**: `if v_neto + round(v_neto*0.19) > 2147483647` → `'La proforma supera el monto máximo facturable'` (neto+iva+total son int4); `v_iva := round(v_neto*0.19)::integer`; correlativo DENTRO de `pg_advisory_xact_lock(hashtextextended('proformas:' || p_empresa::text, 42))`; inserta y marca ODEs.
- **`cambiar_estado_proforma(p_empresa, p_proforma, p_estado, p_motivo default null)`** — rol (ANTES del lookup) → `'Tu rol no permite gestionar proformas'`. Lookup for update; 0 filas → `'La proforma no existe'`. Máquina §3; `'facturada'` o fuera del enum → `'Transición de estado no válida'`. `rechazada` (borrador/enviada/aprobada) exige motivo → `'El rechazo requiere un motivo'` y libera: primero `perform ... order by id for update`, luego `set proforma_id = null where proforma_id = ... and empresa_id = ...`.
- **`facturar_proforma(p_empresa, p_proforma) returns uuid`** — rol → `'Tu rol no permite facturar proformas'`. Lookup for update; `'La proforma no existe'`; exige `'aprobada'` → `'Solo se puede facturar una proforma aprobada'`. Bloquea SUS ODEs (`order by id for update`), re-valida `registrada` → `'Hay una orden no válida, de otro cliente, ya facturada o en otra proforma'` (cinturón-y-tirantes: sin camino natural, se testea con UPDATE de setup o se anota). Inserta `documentos_venta` (`'nota_venta'`, `'borrador'`, cliente y totales DE LA PROFORMA) + línea por ODE: `producto_id = null`, **`descripcion = left('ODE N° ' || o.numero || ' ' || destino_nombre || coalesce(' — guías ' || o.docum, ''), 80)`** (80 = tope NmbItem del DTE SII; evita rechazos cuando llegue SimpleAPI), `cantidad=1`, `precio_neto=o.neto`, `exenta=false`, `subtotal=o.neto`. Marca ODEs y proforma `facturada` + `documento_venta_id`; retorna doc id.
- **`anular_estado_pago(p_empresa, p_documento)`** — rol → `'Tu rol no permite anular estados de pago'`. Lookup del doc for update; 0 filas → `'El documento no existe'`; busca la proforma por `documento_venta_id`: **0 filas → `'El documento no proviene de una proforma'`** (una nota de venta de comercio en borrador NO es deshacible por aquí); luego exige (`tipo='nota_venta' and estado='borrador' and folio is null`) O (`tipo in ('factura','boleta') and estado='rechazado'`) → si no, `'Solo se puede deshacer un estado de pago en borrador o rechazado'`. Revierte: locks ordenados de sus ODEs (`where documento_venta_id = ... and empresa_id = ... order by id for update`), `set estado='registrada', documento_venta_id=null` (conservan `proforma_id`); proforma a `'aprobada'` + `documento_venta_id=null`; borra líneas y documento (definer).

Mensajes exactos (contrato de pgTAP): `'El módulo de transporte no está activo'`, `'Tu rol no permite registrar órdenes de entrega'`, `'Tu rol no permite asignar despachos'`, `'Tu rol no permite anular órdenes de entrega'`, `'Tu rol no permite gestionar proformas'`, `'Tu rol no permite facturar proformas'`, `'Tu rol no permite anular estados de pago'`, `'Cliente no válido'`, `'Destino no válido'`, `'Vehículo no válido'`, `'Conductor no válido'`, `'Ingresa la fecha de ingreso'`, `'Los bultos deben ser un entero mayor a 0'`, `'Los kilos deben ser mayores a 0'`, `'Los kilos superan el máximo admitido'`, `'Los M3 no pueden ser negativos'`, `'Los M3 superan el máximo admitido'`, `'El neto debe ser un entero mayor a 0'`, `'La carga es demasiado pequeña para tarifar'`, `'No se puede asignar despacho a una orden anulada'`, `'La anulación requiere un motivo'`, `'La orden de entrega no existe'`, `'Solo se puede anular una orden registrada'`, `'La orden está en una proforma; recházala primero'`, `'La proforma necesita al menos una orden de entrega'`, `'Hay una orden no válida, de otro cliente, ya facturada o en otra proforma'`, `'La proforma supera el monto máximo facturable'`, `'La proforma no existe'`, `'Transición de estado no válida'`, `'El rechazo requiere un motivo'`, `'Solo se puede facturar una proforma aprobada'`, `'El documento no existe'`, `'El documento no proviene de una proforma'`, `'Solo se puede deshacer un estado de pago en borrador o rechazado'`.

## 10. Criterio de éxito del Plan 11

El transportista puede: activar el módulo con su factor volumétrico, cargar tarifario y flota, digitar una ODE en un minuto (kilo afecto y neto sugerido solos, neto negociable con regla de re-sugerencia clara), asignar el despacho cuando salga el camión, armar la proforma idéntica a la que hoy emite (PF-N°, nómina completa, IVA sobre el neto total), enviarla/imprimirla/aprobarla y facturarla al SII — con salida de CUALQUIER estado (rechazo libera incluso aprobadas, deshacer revierte borradores y DTE rechazados); ninguna carga válida revienta con errores crudos (mínimos, topes y overflow con mensajes claros); nada se factura dos veces ni deadlockea; el top de productos no se contamina; comercio puro no ve ni usa nada por API; y de pasada quedan cerrados el hueco de `empresas` y tres debilidades de emisión.
